Get up and running with Pixi Build Backends in minutes. This guide walks you through creating your first buildable package using the Python backend, then shows you how to explore other backends.

<Warning>
`pixi-build` is a preview feature and will change until it is stabilized. You must opt in by adding "pixi-build" to `workspace.preview`:

```toml
[workspace]
preview = ["pixi-build"]
```
</Warning>

## Prerequisites

Before you begin, make sure you have:

- [Pixi](https://pixi.sh) installed (version 0.40.0 or later)
- A project you want to package (or follow along with our examples)

## Your First Build: Python Package

<Steps>

<Step title="Create a Python project with pyproject.toml">

Let's start with a simple Python package. Create a new directory and add a `pyproject.toml`:

```toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "my-awesome-package"
version = "1.0.0"
description = "My first Pixi-built package"
requires-python = ">=3.8"
dependencies = [
    "requests>=2.28",
]
```

Add a simple Python module in `my_awesome_package/__init__.py`:

```python
def hello():
    return "Hello from my-awesome-package!"
```

</Step>

<Step title="Add build backend configuration to pixi.toml">

Create a `pixi.toml` file that tells Pixi how to build your package:

```toml
[workspace]
name = "my-awesome-package"
version = "1.0.0"
channels = ["https://prefix.dev/conda-forge"]
platforms = ["linux-64", "osx-64", "osx-arm64", "win-64"]
preview = ["pixi-build"]

[package]
name = "my-awesome-package"
version = "1.0.0"

[package.build]
backend = { name = "pixi-build-python", version = "*" }
channels = ["https://prefix.dev/conda-forge"]
```

The `[package.build]` section is where the magic happens:
- `backend.name` specifies which build backend to use (`pixi-build-python` for Python projects)
- `backend.version` specifies the backend version (use `"*"` for latest)
- `channels` tells Pixi where to find the backend package

</Step>

<Step title="Build your package">

Build your package into a conda package:

```bash
pixi build
```

This command:
1. Downloads and installs the `pixi-build-python` backend
2. Reads your `pyproject.toml` and `pixi.toml` configuration
3. Generates a rattler-build recipe
4. Builds a conda package (`.conda` file)

The built package will be in `.pixi/build/` directory.

<Tip>
Use `pixi build --verbose` to see detailed output of what's happening during the build process.
</Tip>

</Step>

<Step title="Install and test your package">

Install your newly built package in a Pixi environment:

```bash
pixi add --manifest-path pixi.toml my-awesome-package
pixi install
```

Test it works:

```bash
pixi run python -c "import my_awesome_package; print(my_awesome_package.hello())"
```

</Step>

</Steps>

## Understanding the Build System Section

The `[package.build]` section in your `pixi.toml` controls how your package is built:

```toml
[package.build]
# Required: which backend to use
backend = { name = "pixi-build-python", version = "*" }

# Required: where to find the backend
channels = ["https://prefix.dev/conda-forge"]

# Optional: backend-specific configuration
[package.build.config]
noarch = true  # Build platform-independent package
env = { SETUPTOOLS_SCM_PRETEND_VERSION = "1.0.0" }
```

Each backend has its own configuration options. See the [backend documentation](/backends/python) for details.

## Common Build Commands

Here are the most common commands you'll use:

```bash
# Build all packages in your workspace
pixi build

# Build a specific package
pixi build --package my-package

# Build with verbose output
pixi build --verbose

# Build for a specific platform
pixi build --target-platform linux-64
```

## Quick Start with Other Backends

### Rust Package

For a Rust project with `Cargo.toml`:

```toml
[package.build]
backend = { name = "pixi-build-rust", version = "*" }
channels = ["https://prefix.dev/conda-forge"]
```

The Rust backend automatically reads metadata from your `Cargo.toml` and builds your project using `cargo install`.

### CMake C++ Project

For a CMake-based C++ project:

```toml
[package.build]
backend = { name = "pixi-build-cmake", version = "*" }
channels = ["https://prefix.dev/conda-forge"]

[package.build.config]
compilers = ["c", "cxx"]
extra-args = ["-DCMAKE_BUILD_TYPE=Release"]
```

The CMake backend uses Ninja and automatically configures compilers for your platform.

### Using rattler-build Recipes

If you need full control, use the rattler-build backend with a `recipe.yaml`:

```toml
[package.build]
backend = { name = "pixi-build-rattler-build", version = "*" }
channels = ["https://prefix.dev/conda-forge"]
```

This backend looks for a `recipe.yaml` file in your project root and builds it directly with [rattler-build](https://rattler.build).

## Troubleshooting

<AccordionGroup>

<Accordion title="Build fails with 'backend not found'">
Make sure you've added the correct channels:

```toml
[package.build]
channels = ["https://prefix.dev/conda-forge"]
```

You can also add the pixi-build-backends channel for the latest versions:

```toml
[package.build]
channels = [
  "https://prefix.dev/pixi-build-backends",
  "https://prefix.dev/conda-forge"
]
```
</Accordion>

<Accordion title="Want to see the generated recipe?">
The build backend generates a complete rattler-build recipe during the build process. You can find it in:

```
.pixi/build/work/<package-name>--<hash>/debug/recipe.yaml
```

This is useful for debugging build issues or understanding how your package is being built.

To rebuild using the generated recipe directly:

```bash
rattler-build build --recipe .pixi/build/work/<package-name>--<hash>/debug/
```
</Accordion>

<Accordion title="How do I add build dependencies?">
Use the `[package.build-dependencies]` section for build-time dependencies:

```toml
[package.build-dependencies]
cmake = ">=3.20"
ninja = "*"

[package.host-dependencies]
python = "3.11"
hatchling = "*"

[package.run-dependencies]
requests = ">=2.28"
```

See [Dependency Types](https://pixi.sh/latest/build/dependency_types/) for more information.
</Accordion>

<Accordion title="Python: Dependencies from pyproject.toml not being used?">
By default, automatic PyPI-to-conda dependency mapping is disabled. To enable it:

```toml
[package.build.config]
ignore-pypi-mapping = false
```

This will automatically map dependencies from your `pyproject.toml` to conda packages.
</Accordion>

</AccordionGroup>
